# LLM Inference 推理

## 了解基礎

推理（Inference）指的是使用訓練好的模型（LLM），根據給定的 Input Prompt（輸入提示），產生類似人類語言文本的過程。模型利用訓練中累積的知識逐字逐句地產生回應，從數十億個參數學習到的機率來預測下一個 token，這也是為什麼模型能夠產生上下文連貫內容的原因。

---

## 注意力（Attention）的作用

注意力機制賦予模型理解 Context 並產生連貫 Response 的能力。比如 "法國的首都是 ..."，句子中「法國」和「首都」這兩個詞對於下一個正確的詞應該是「巴黎」至關重要。

> **注意力機制**是語言模型能夠產生連貫且有上下文感知文本的關鍵能力。

---

## Context Length 與成本

Context Length 指模型一次能接收的最大 token 數，這個數字背後的代價不是線性的：

- **記憶體佔用**：隨 context 長度**二次方**成長（n²）— 長度翻倍，記憶體暴增四倍
- **處理速度**：隨 context 長度**線性**下降

Context Length 受以下因素影響：
1. 模型的架構跟尺寸
2. 可用的運算資源
3. 輸入跟預期輸出的複雜性

這就是為什麼模型的 context window 是一個很現實的瓶頸，不是不想做大，是硬體跟成本會爆炸。

> 精心設計的 **提示詞（prompt）** 可以引導 LLM 產生符合預期的結果

## 兩階段推理過程（The Two-Phase Inference Process）

兩階段分別是 **預填充（Prefill Phase）** 跟 **解碼（Decode Phase）**。

### Prefill Phase

這階段就像烹飪一樣，所有的食材都在這裡加工跟準備，主要分為 3 個步驟：

1. **分詞（Tokenization）**：將原始內容轉換成模型理解的 token
2. **嵌入轉換（Embedding Conversion）**：將 token 轉換成能夠捕捉其意義的數值表示
3. **初步處理（Initial Processing）**：將向量（vector）輸入至神經網絡，深入理解上下文

> 這個階段的計算量最大，因為模型需要一次處理所有輸入的 token，就像我們回信時，先閱讀並理解一整段文字。

### Decode Phase

這個階段是實際產生文字的階段。模型每生成一個 token，稱為所謂的 **自回歸過程（autoregressive process）**，其中每個新 token 都依賴先前的 token。

步驟有：
1. **注意力計算（Attention Computation）**：回顧所有先前的 token 以理解上下文
2. **機率計算（Probability Calculation）**：決定下一個可能出現的 token 的機率
3. **Token Selection**：根據這些機率選擇下一個 token
4. **持續性檢查**：決定是否繼續或停止生成

> 這個階段會佔用大量記憶體，因為模型需要追蹤先前所有產生的 token 以及 token 之間的關係

## 取樣策略 Sampling Strategies

模型計算出每個可能 token 的機率之後，怎麼「選」下一個 token 其實大有學問，不同策略會帶來截然不同的輸出風格。

### Temperature（溫度）

把 temperature 想像成「創意旋鈕」：
- **Temperature 高（> 1.0）**：機率分佈被「攤平」，冷門選項也有機會被選到，輸出更有創意但也更不可預測
- **Temperature 低（< 1.0）**：機率差距被放大，模型更傾向選最有把握的，輸出更保守穩定
- **Temperature = 0**：每次都選機率最高的，輸出完全確定性

### Top-k Sampling

只從機率最高的 k 個 token 裡面挑，其他直接忽略。像是考試只考前 k 名的範圍，確保不會選到太離譜的答案。

### Top-p Sampling（又稱 Nucleus Sampling）

不限定固定幾個，而是把機率由高到低累加，累積到 p（例如 0.9）就停，在這個範圍內取樣。好處是：如果有個明顯最佳答案，候選集就很小；如果沒有，候選集就自動變大。

### Beam Search

不是只追一條路，而是同時維持多條候選路徑（beam width），最後選整體機率最高的那條。生成更連貫，但計算成本也更高。

## KV Cache

Decode 階段每生成一個 token，都要對「所有之前的 token」做 Attention 計算——這表示前面算過的東西每次都要重算一遍，非常浪費。

**KV Cache** 的做法是把 Attention 計算中的 Key 跟 Value 矩陣快取起來，下一個 token 直接拿來用，不用重算。

- 優點：大幅加速 Decode 速度
- 代價：佔更多記憶體（VRAM）

這個技術幾乎是所有現代 LLM 推理框架的標配，是讓長文生成不爆炸的關鍵之一。

## 推理效能指標

實際部署 LLM 時，有四個關鍵數字你一定會碰到：

| 指標 | 全名 | 意義 |
|------|------|------|
| **TTFT** | Time to First Token | 第一個字多快出來，直接影響使用者體感 |
| **TPOT** | Time Per Output Token | 後續每個 token 的生成速度，決定整體產出速度 |
| **Throughput** | — | 同時能處理幾個 request，影響系統的擴展能力和成本 |
| **VRAM Usage** | — | GPU 記憶體需求，通常是最先碰到的硬體瓶頸 |

這四個指標之間往往是 trade-off：想要 TTFT 快，通常要犧牲 Throughput；想要處理更長的 context，VRAM 就會暴增。沒有魔法，只有取捨。
