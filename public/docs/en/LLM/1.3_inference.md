# LLM Inference

## Understanding the Basics

Inference refers to the process of using a trained model (LLM) to generate human-like text based on a given input prompt. The model produces responses token by token, leveraging knowledge accumulated during training — predicting the next token based on probabilities learned from billions of parameters. This is how models are able to generate contextually coherent content.

---

## The Role of Attention

The attention mechanism gives a model the ability to understand context and generate coherent responses. For example, in the sentence "The capital of France is ...", the words "France" and "capital" are critical for the model to correctly predict the next word as "Paris".

> **The attention mechanism** is the core capability that enables language models to produce coherent, context-aware text.

---

## Context Length and Cost

Context length refers to the maximum number of tokens a model can process at once. The cost behind this number is not linear:

- **Memory usage**: grows **quadratically** with context length (n²) — doubling the length quadruples memory consumption
- **Processing speed**: degrades **linearly** as context length increases

Context length is affected by:
1. Model architecture and size
2. Available compute resources
3. Complexity of the input and expected output

This is why a model's context window is a very real bottleneck — it's not that no one wants larger windows, it's that the hardware and cost would spiral out of control.

> A well-crafted **prompt** can guide an LLM toward the desired output.

## The Two-Phase Inference Process

The two phases are the **Prefill Phase** and the **Decode Phase**.

### Prefill Phase

This phase is like cooking — all ingredients are prepared and processed here. It consists of 3 main steps:

1. **Tokenization**: Converts raw input into tokens the model can understand
2. **Embedding Conversion**: Transforms tokens into numerical representations that capture their meaning
3. **Initial Processing**: Feeds the vectors into the neural network for deep contextual understanding

> This phase is the most computationally intensive, as the model must process all input tokens at once — similar to reading and understanding an entire message before composing a reply.

### Decode Phase

This is the phase where text is actually generated. The model produces one token at a time in what's called an **autoregressive process**, where each new token depends on all previous tokens.

The steps are:
1. **Attention Computation**: Reviews all prior tokens to understand context
2. **Probability Calculation**: Determines the probability distribution over possible next tokens
3. **Token Selection**: Selects the next token based on those probabilities
4. **Continuation Check**: Decides whether to keep generating or stop

> This phase consumes significant memory, as the model must track all previously generated tokens and the relationships between them.

## Sampling Strategies

Once the model has computed probabilities for every possible next token, *how* it selects the next one is a nuanced decision — different strategies produce very different output styles.

### Temperature

Think of temperature as a "creativity dial":
- **High temperature (> 1.0)**: Flattens the probability distribution, giving less likely tokens a better chance of being selected — outputs become more creative but also less predictable
- **Low temperature (< 1.0)**: Amplifies probability differences, making the model more likely to pick its top choice — outputs become more conservative and stable
- **Temperature = 0**: Always selects the highest-probability token — fully deterministic output

### Top-k Sampling

Only sample from the top k tokens by probability, ignoring all others. It's like an exam that only covers the top k candidates — ensuring the model never picks something completely off-base.

### Top-p Sampling (aka Nucleus Sampling)

Rather than a fixed count, accumulate probabilities from highest to lowest until the total reaches p (e.g., 0.9), then sample within that set. The benefit: if there's one clearly dominant answer, the candidate pool stays small; if not, it expands automatically.

### Beam Search

Instead of following a single path, maintain multiple candidate sequences simultaneously (beam width), then select the one with the highest overall probability at the end. This produces more coherent outputs, but at a higher computational cost.

## KV Cache

During the Decode phase, every new token requires an attention computation over *all previous tokens* — meaning everything computed before gets recalculated from scratch each time, which is extremely wasteful.

**KV Cache** solves this by caching the Key and Value matrices from the attention computation, so the next token can reuse them directly without recomputation.

- Benefit: Dramatically speeds up the Decode phase
- Trade-off: Consumes more memory (VRAM)

This technique is standard in virtually every modern LLM inference framework and is one of the key reasons long-form generation doesn't grind to a halt.

## Inference Performance Metrics

When deploying an LLM in production, there are four key numbers you'll always encounter:

| Metric | Full Name | Meaning |
|--------|-----------|---------|
| **TTFT** | Time to First Token | How quickly the first token appears — directly impacts perceived responsiveness |
| **TPOT** | Time Per Output Token | Generation speed for each subsequent token — determines overall output throughput |
| **Throughput** | — | How many requests can be handled concurrently — affects scalability and cost |
| **VRAM Usage** | — | GPU memory requirement — typically the first hardware bottleneck you'll hit |

These four metrics are inherently trade-offs: optimizing for lower TTFT usually hurts Throughput; supporting longer context windows causes VRAM to skyrocket. There's no magic — only trade-offs.
